# Migration Notes

These notes document an ad-hoc migration from an existing Ollie/Hermes install on one host to a fresh one (e.g., AWS EC2 → Hetzner Cloud). The 6 install scripts handle a **fresh install with no data carryover**. This doc handles the data carryover.

> **Not scripted in v1 by design.** Migration is a one-time, host-specific operation. Generalizing it into a script invites edge cases (mismatched versions, divergent profile sets, custom plugins). When you need to migrate, follow the playbook below.

## What gets migrated

| What | From | To | Why |
|---|---|---|---|
| `~/.hermes/auth.json` | source host | new host | Codex/Anthropic OAuth tokens — avoids re-login |
| `~/.hermes/config.yaml` | source | new | Model + provider + MCP server config (default profile) |
| `~/.hermes/state.db` | source | new | Default profile's session/transcript history |
| `~/.hermes/profiles/<name>/config.yaml` | source | new | Per-profile MCP servers, model overrides |
| `~/.hermes/profiles/<name>/state.db` | source | new | Per-profile sessions |
| Cortex volume `/data` (memory + brain) | source container | new container | Memory thoughts + brain knowledge files |
| `TELEGRAM_*` lines from `~/.hermes/.env` | source | appended to new | Bot token + allowlist |

## What does NOT migrate (and why)

- **`API_SERVER_KEY` and other host-generated secrets** — the new host has its own (auto-generated by `02-install-hermes.sh`). Mixing them breaks dashboard auth to the gateways.
- **Per-profile `.env` files (other than `TELEGRAM_*`)** — they only contain `API_SERVER_*` which the install scripts already wrote correctly on the new host.
- **`ORCHESTRATOR_KEY`** — new host generates its own.
- **Docker images** — pull fresh on the new host (`06-install-stack.sh` does this).
- **systemd unit files** — install scripts write them with the latest hardened template.
- **profile dirs other than `config.yaml` + `state.db`** — they contain a lot of cache/regenerable data (~50MB). Hermes will rebuild on first start.

## Playbook (~10 min for small installs)

Run from a workstation that has SSH access to both hosts. Replace `<SRC>` and `<NEW>` with the actual IPs/keys.

### 1. Stage on source host

```bash
ssh -i <SRC_KEY> ubuntu@<SRC_IP> '
  mkdir -p /tmp/migration
  cd ~
  tar czf /tmp/migration/hermes-state.tar.gz \
    .hermes/auth.json \
    .hermes/config.yaml \
    .hermes/state.db \
    .hermes/profiles/*/config.yaml \
    .hermes/profiles/*/state.db
  docker exec cortex tar -C /data -czf - . > /tmp/migration/cortex-data.tar.gz
  grep -E "^TELEGRAM_" ~/.hermes/.env > /tmp/migration/telegram.env
  ls -la /tmp/migration/
'
```

### 2. Transfer through your workstation

```bash
mkdir -p /tmp/migration
scp -i <SRC_KEY> -r ubuntu@<SRC_IP>:/tmp/migration/* /tmp/migration/
scp -i <NEW_KEY> -r /tmp/migration/* ubuntu@<NEW_IP>:/tmp/migration/
```

### 3. Apply on new host (after fresh install scripts have run)

```bash
ssh -i <NEW_KEY> ubuntu@<NEW_IP> '
  set -e
  echo "stopping services..."
  systemctl --user stop hermes-gateway hermes-gateway-* hermes-dashboard hermes-dashboard-*
  docker compose -f ~/hermes-stack/docker-compose.yml stop cortex

  echo "applying hermes state..."
  cd ~
  tar xzf /tmp/migration/hermes-state.tar.gz

  echo "appending TELEGRAM_* to .env..."
  sed -i "/^TELEGRAM_/d" ~/.hermes/.env
  cat /tmp/migration/telegram.env >> ~/.hermes/.env

  echo "applying cortex volume..."
  rm -rf /tmp/cortex-staging && mkdir -p /tmp/cortex-staging
  tar xzf /tmp/migration/cortex-data.tar.gz -C /tmp/cortex-staging
  docker run --rm \
    -v hermes-stack_cortex-data:/data \
    -v /tmp/cortex-staging:/source \
    alpine sh -c "rm -rf /data/* 2>/dev/null; cp -a /source/. /data/"

  echo "restarting..."
  systemctl --user start hermes-gateway hermes-gateway-*
  sleep 2
  systemctl --user start hermes-dashboard hermes-dashboard-*
  docker compose -f ~/hermes-stack/docker-compose.yml start cortex
'
```

### 4. Verify on the new host

- All systemd services `active`
- All dashboards respond `200` on their ports
- `hermes mcp test <name>` for each profile shows `✓ Connected` with tool count
- Browser to `http://<NEW_IP>:3000/chat` — chat with each agent
- Memory tab shows existing thoughts
- Transcripts tab shows historical sessions

### 5. Resolve the Telegram polling conflict

While both source and new hosts are running, Hermes' Telegram bot polling will fight (the bot token can only be polled by one instance). Pick one to stop:

```bash
# Stop on the OLD host:
ssh ubuntu@<SRC_IP> 'systemctl --user stop hermes-gateway'
```

Within ~30 seconds the new host's gateway will win polling cleanly.

### 6. Decommission the source host

After you've verified the new host for at least a day (so any cron'd skills have fired, you've used both agents, etc.):

- Stop the source host's services
- Snapshot the source disk (one-time insurance)
- Terminate / delete the source instance

## What could go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| Chat returns 401 from dashboard | New host's `API_SERVER_KEY` not in `~/hermes-stack/.env` | Re-run `06-install-stack.sh` (it derives keys from the host config) |
| Empty Memory tab | Cortex volume not actually loaded | Re-run the `docker run --rm -v hermes-stack_cortex-data:/data ...` step |
| Telegram bot ignores DMs | Polling conflict with the old host | Stop the old host's gateway (step 5) |
| `mcp test agentmail` succeeds but agents say "no inbox" | Stale tool cache in the agent's session | Start a NEW chat session (existing ones miss the post-migration MCP) |
| Gateway logs warn "auth refresh failed" | Codex tokens copied but the gateway restart raced with token use | Re-auth with `hermes auth add openai-codex --type oauth --no-browser --manual-paste` (the old `hermes login` was removed in v0.16+), then `systemctl --user restart hermes-gateway`. Check with `hermes auth status openai-codex`. |
